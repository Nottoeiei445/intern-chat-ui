1. การวางโครงสร้างแบบ "Service Layer" (แยกหน้าที่ให้ชัดเจน)
มือใหม่มักทำ: เขียนคำสั่งยิง API (fetch) ไว้ข้างในปุ่มกดหรือในหน้าเว็บตรงๆ
ปัญหา: พอแอปใหญ่ขึ้น มีหลายหน้าต้องยิง API เดิมซ้ำๆ ถ้าหลังบ้านเปลี่ยน URL เราต้องไล่แก้ทุกหน้า (เหนื่อยและพลาดง่าย)
วิธีที่ใช้จัดการ:
เราใช้หลักการ Separation of Concerns โดยการแยกไฟล์ออกเป็น 2 ส่วน คือ
-Services: เปรียบเหมือนคนส่งของ มีหน้าที่ยิง API และจัดการข้อมูลให้พร้อมใช้เท่านั้น
-Hooks/Components: เปรียบเหมือนคนใช้งาน มีหน้าที่เรียกใช้คนส่งของมาแสดงผลบนจอ
ผลลัพธ์: ถ้าหลังบ้านเปลี่ยนทางเดินรถ (URL) เราแก้ที่ไฟล์ service.ts ที่เดียว หน้าเว็บทุกหน้าจะทำงานได้ปกติทันที

2. Hybrid API Client เลือกเครื่องมือให้เหมาะกับงาน
สิ่งที่ได้เรียนรู้: เครื่องมือแต่ละอย่างมีจุดเด่นไม่เหมือนกัน เราจึงเลือกใช้ 2 ท่าควบคู่กัน คือ
-Native Fetch Wrapper (api-client.ts): เราสร้าง "ตัวยิง API รุ่นไลท์เวท" ขึ้นมาเองโดยใช้ fetch ที่มีมากับเบราว์เซอร์ เพื่อใช้ในฟีเจอร์ Chat และ Map
เหตุผล: เพื่อลดการพึ่งพา Library ภายนอก และทำให้เราเข้าใจโครงสร้าง HTTP Header และ Error Handling อย่างลึกซึ้ง
-Axios Interceptors (auth.service.ts): เราใช้ axios เฉพาะในระบบยืนยันตัวตน (Auth)
เหตุผล: เพราะ Axios มีระบบ ด่านตรวจ (Interceptors) ที่ฉลาดมาก เหมาะกับลอจิกที่ซับซ้อนอย่างการต่ออายุตั๋ว (Token Refresh)
